# Физическая модель базы данных (уточнённый вариант)

Ниже приведено **пересмотренное описание физической модели**, с учётом:
- разделения медиапотоков на **видео и аудио**;
- **осознанного сохранения JSON** в отчётах для поддержки dashboard-панели;
- соблюдения **3НФ для транзакционных данных** при допустимых денормализациях в аналитическом слое.

Модель ориентирована на **meeting-platform (онлайн)** с полной поддержкой **оффлайн-сценариев**.

---

## 1. Человек

**Назначение:** базовая сущность физического лица.

### person
- person_id (PK)
- last_name
- first_name
- middle_name
- created_at

---

## 2. Профиль участника

**Назначение:** HR-профиль человека внутри организации.

### profile
- profile_id (PK)

- employee_number (UNIQUE, NULLABLE)
- position
- department
- profile_status_id (FK → profile_status.profile_status_id)
- hiring_date
- resume_url

### profile_status (справочник)
- profile_status_id (PK)
- code (active, archived, recruiting)
- description

---

## 3. Типы сеансов и роли

## Архитектурные пояснения

### Почему видео и аудио разделены
- разные пайплайны обработки;
- разная частота и семантика данных;
- упрощение масштабирования (отдельный ASR / CV сервис).

### Почему JSON допустим в отчётах
- отчёт — **не первичный факт**, а агрегат;
- dashboard требует быстрого парсинга;
- JSON — классический **CQRS / Read Model**;
- транзакционные данные остаются нормализованными.

### Итог
- **3НФ соблюдена** для ядра данных;
- **денормализация допустима** на уровне аналитики;
- модель масштабируема и академически корректна;
- структура хорошо ложится на PostgreSQL + BI.
